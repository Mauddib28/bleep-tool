# GATT Enumeration Commands

BLEEP provides two CLI commands for enumerating a BLE device's GATT database after connecting: **`gatt-enum`** and **`enum-scan`**.  They share the same connection and service-discovery pipeline (`connect_and_enumerate__bluetooth__low_energy` in `bleep/ble_ops/connect.py`) but serve different purposes.

- **`gatt-enum`** — Single-pass GATT read with optional deep-retry mode.
- **`enum-scan`** — Multi-variant enumeration engine layering increasingly aggressive techniques on top of the base enumeration.

Both commands output a human-readable tree via `format_gatt_tree()` (`bleep/ble_ops/conversion.py`) with UUID name resolution, hex/ASCII values, and permission/landmine map summaries.  When device-level D-Bus properties are available, a **Device Information** block is prepended showing ManufacturerData (key as hex+decimal, value as Hex/ASCII), ServiceData (UUID with resolved name), Class (decoded), Appearance, RSSI, TxPower, Modalias, Bonded, WakeAllowed, Icon, AdvertisingFlags, AdvertisingData, Battery1 (Percentage/Source), Input1 (ReconnectMode), and advertised UUIDs.  Non-printable bytes in ASCII output are rendered as the Unicode replacement character (U+FFFD).

The GATT tree itself now displays additional per-object properties:

- **Services**: Handle, Primary/Secondary designation, and Includes (referenced included services).
- **Characteristics**: Handle, Flags/Properties, MTU (when available post-connection), Notifying state, and Hex/ASCII value lines.
- **Descriptors**: Handle, Flags (read/write permissions), and Hex/ASCII value lines.

See [BlueZ D-Bus Interface Property Reference](bluez_interface_properties.md) for full details on each property's significance and security implications.

## `gatt-enum` — Single-Pass GATT Enumeration

### Synopsis

```bash
bleep gatt-enum <ADDRESS> [--deep] [--report]
```

### Arguments

| Argument | Required | Description |
|----------|----------|-------------|
| `ADDRESS` | Yes | Target Bluetooth MAC address (e.g. `CC:50:E3:B6:BC:A6`) |
| `--deep` | No | Perform deep enumeration with retry reads and descriptor probing |
| `--report` | No | Print landmine & security reports as JSON instead of the GATT tree |

### Standard Mode (no flags)

Connects to the target, discovers the GATT service/characteristic structure, and performs a **single `read_value()` call** per readable characteristic.

Key behaviours:

- Only characteristics with the `"read"` property flag are probed.
- Descriptors are read with a single call (no retry).
- The mapping uses **lowercase keys** (`chars`, `properties`, `handle`, `value`, `raw`).
- On read failure, the error is classified into permission/landmine maps but no retry occurs.
- Results are persisted to the observation database (services and characteristics upserted).

Implementation path: `cli.py` → `connect_and_enumerate__bluetooth__low_energy(deep_enumeration=False)` → `device.services_resolved(deep=False)` → `_enumerate_gatt_values(deep=False)` in `bleep/dbuslayer/device_le.py`.

### Deep Mode (`--deep`)

Connects and then **rebuilds the entire mapping from scratch** using retry logic and broader probing.

Key behavioural differences from standard mode:

| Aspect | Standard | Deep |
|--------|----------|------|
| Read strategy | Single `read_value()` | `safe_read_with_retry()` — 3 attempts, 0.3 s delay between retries |
| Characteristics probed | Only those with `"read"` flag | Those with `"read"` **or** `"write"` flag |
| Descriptor reads | Single call, no retry | `safe_read_with_retry()` with retries |
| Mapping construction | In-place update of existing mapping | Full rebuild from scratch (`ble_device__mapping = {}`) |
| Key casing in output | Lowercase (`chars`, `value`, `handle`, `properties`) | Uppercase (`Characteristics`, `Value`, `Handle`, `Flags`, `Raw`, `Descriptors`) |
| Descriptor inclusion | Only when descriptors have values | Always present (empty `{}` if none) |
| Flag format | `"properties": {"read": true, ...}` (dict) | `"Flags": ["read", "write", ...]` (list) |

Implementation path: same as standard but with `deep_enumeration=True` → `_enumerate_gatt_values(deep=True)`.

### Report Mode (`--report`)

When `--report` is passed, the command outputs JSON containing:

```json
{
  "landmine_report": { ... },
  "security_report": { ... }
}
```

These reports are generated by `device.get_landmine_report()` and `device.get_security_report()`.  This flag is mutually exclusive with tree output — when `--report` is active, no GATT tree is printed and database persistence of characteristics is skipped.

### Examples

```bash
# Standard single-pass enumeration (tree output)
bleep gatt-enum CC:50:E3:B6:BC:A6

# Deep enumeration with retry reads and descriptor probing
bleep gatt-enum CC:50:E3:B6:BC:A6 --deep

# Landmine and security reports as JSON
bleep gatt-enum CC:50:E3:B6:BC:A6 --report

# Deep enumeration with reports
bleep gatt-enum CC:50:E3:B6:BC:A6 --deep --report
```

## `enum-scan` — Multi-Variant Enumeration Engine

### Synopsis

```bash
bleep enum-scan <ADDRESS> [--variant passive|naggy|pokey|brute] [options]
```

### Arguments

| Argument | Required | Default | Description |
|----------|----------|---------|-------------|
| `ADDRESS` | Yes | — | Target Bluetooth MAC address |
| `--variant` | No | `passive` | Enumeration strategy: `passive`, `naggy`, `pokey`, or `brute` |
| `--rounds` | No | `3` | Number of rounds for `pokey` variant |
| `--write-char` | Brute only | — | Characteristic UUID to target (or `all` for all writable) |
| `--range` | No | — | Hex start-end for brute payload range (e.g. `00-FF`) |
| `--patterns` | No | — | Comma-separated payload patterns (see below) |
| `--payload-file` | No | — | Path to a binary payload file |
| `--force` | No | — | Ignore landmine/permission maps for brute writes |
| `--verify` | No | — | Read back after each brute write to detect changes |
| `--controlled` | No | — | Use `EnumerationController` for structured multi-attempt enumeration with error annotations |

### Variant Details

#### `passive` (default)

Functionally equivalent to `gatt-enum` without `--deep`.  Connects, enumerates the GATT database with single reads, and returns.

Implementation: `bleep/ble_ops/scan.py::passive_enum()` → `_base_enum()` → `connect_and_enumerate__bluetooth__low_energy(deep_enumeration=False)`.

```bash
bleep enum-scan CC:50:E3:B6:BC:A6
bleep enum-scan CC:50:E3:B6:BC:A6 --variant passive
```

#### `naggy`

After the initial enumeration, performs **3 rounds of multi-reads** across all readable characteristics using `multi_read_all()`.  Detects characteristics whose values **changed across rounds** and flags them in the output tree with change indicators.

Returns additional data:
- `multi_read` — dict of round → characteristic → value
- `changed_chars` — set of characteristic labels whose values differed across rounds

Use case: identify transient or dynamic characteristics (counters, timestamps, sensor data).

Implementation: `bleep/ble_ops/scan.py::naggy_enum()`.

```bash
bleep enum-scan CC:50:E3:B6:BC:A6 --variant naggy
```

#### `pokey`

Performs **N rounds** (default 3, configurable via `--rounds`) of enumeration.  After each round, performs **light write probes** — writes `0x00` / `0x01` to every writable characteristic via `small_write_probe()`.  Skips descriptors.

Use case: observe device reactions to minimal writes; detect state-changing characteristics without heavy fuzzing.

Implementation: `bleep/ble_ops/scan.py::pokey_enum()`.

```bash
bleep enum-scan CC:50:E3:B6:BC:A6 --variant pokey
bleep enum-scan CC:50:E3:B6:BC:A6 --variant pokey --rounds 5
```

#### `brute`

**Requires `--write-char`.**  After the initial enumeration, performs brute-force write operations against a specific characteristic UUID (or `all` writable characteristics).  Payloads are generated from one or more of: hex ranges, pattern generators, or raw binary files.

By default, brute mode **respects landmine and permission maps** — it refuses writes to characteristics flagged as dangerous.  Use `--force` to override this safety check.

Implementation: `bleep/ble_ops/scan.py::brute_enum()` → `brute_write_range()` or `multi_write_all()`.

**Payload sources** (combinable):

| Source | Flag | Example |
|--------|------|---------|
| Hex range | `--range AA-BB` | `--range 00-FF` generates single bytes 0x00 through 0xFF |
| Pattern generators | `--patterns <list>` | `--patterns ascii,inc,alt` |
| Binary file | `--payload-file <path>` | `--payload-file fuzz.bin` |

**Available pattern keywords:**

| Pattern | Description |
|---------|-------------|
| `ascii` | All printable ASCII bytes |
| `inc` | Incrementing length-prefixed byte strings |
| `alt` | Single 0xAA / 0x55 bytes |
| `repeat:<byte>:<len>` | Repeat a byte value N times (e.g. `repeat:ff:4`) |
| `hex:<hexstring>` | Arbitrary hex string as raw bytes (e.g. `hex:deadbeef`) |

```bash
# Brute-force a single characteristic with default byte range
bleep enum-scan CC:50:E3:B6:BC:A6 --variant brute --write-char 0000ff01-0000-1000-8000-00805f9b34fb

# Brute-force all writable characteristics, verify reads, ignore safety
bleep enum-scan CC:50:E3:B6:BC:A6 --variant brute --write-char all --verify --force

# Custom range + patterns + payload file
bleep enum-scan CC:50:E3:B6:BC:A6 --variant brute \
  --write-char 0000ff01-0000-1000-8000-00805f9b34fb \
  --range 00-0F \
  --patterns ascii,repeat:ff:4 \
  --payload-file custom_fuzz.bin \
  --verify
```

### Controlled Mode (`--controlled`)

When `--controlled` is passed, `enum-scan` uses the `EnumerationController` (`bleep/ble_ops/enum_controller.py`) instead of the standard dispatch.  This provides:

- Structured multi-attempt enumeration (up to 3 attempts by default)
- Automatic reconnection handling via `ReconnectionMonitor`
- Error annotations with typed error classifications
- JSON-formatted output of the `EnumerationResult` data

The `--controlled` flag works with any `--variant` value — the variant name is passed as the `mode` parameter to `EnumerationController.enumerate()`.

```bash
bleep enum-scan CC:50:E3:B6:BC:A6 --controlled
bleep enum-scan CC:50:E3:B6:BC:A6 --variant naggy --controlled
```

## Comparison: `gatt-enum` vs `enum-scan`

| Aspect | `gatt-enum` | `enum-scan` |
|--------|-------------|-------------|
| Purpose | Single-pass GATT read | Multi-strategy enumeration toolkit |
| Variants | None (only `--deep`) | `passive`, `naggy`, `pokey`, `brute` |
| Write operations | Never writes | `pokey` (light probes), `brute` (full write sweeps) |
| Multi-round reads | No | `naggy` (3 rounds), `pokey` (N rounds) |
| Change detection | No | `naggy` detects value changes across rounds |
| `--report` flag | Yes (landmine/security reports) | No |
| `--deep` flag | Yes | No (all variants use `deep=False` internally) |
| `--controlled` mode | No | Yes (`EnumerationController`) |
| Brute-force writing | No | Yes (`--write-char`, `--range`, `--patterns`, `--payload-file`) |
| DB persistence | Inline upsert in CLI handler | Via `_persist_mapping()` helper in `scan.py` |
| Output extras | `device_name` in tree header | `changed_chars` highlighting in tree (naggy) |

### When to Use Which

- **`gatt-enum`** — First-look read-only enumeration of a device.  Use `--deep` when the device has flaky characteristics that need retry logic, or when you want descriptor details.  Use `--report` for security assessments.
- **`enum-scan --variant passive`** — Equivalent to `gatt-enum` without `--deep`.  Useful when scripting with other `enum-scan` variants.
- **`enum-scan --variant naggy`** — When you need to detect dynamic or changing characteristics (e.g. counters, sensor data, timestamps).
- **`enum-scan --variant pokey`** — When you want to observe device reactions to minimal writes before committing to heavier probing.
- **`enum-scan --variant brute`** — Active fuzzing of writable characteristics with configurable payloads.

### Troubleshooting: `br-connection-profile-unavailable` on dual-mode devices

`gatt-enum` calls `Device1.Connect()` which requires BlueZ to have a registered
profile handler for at least one of the remote device's advertised services.
Audio-primary devices (phones, speakers, headsets) mainly advertise A2DP, HFP,
and HSP — if the host has **no audio stack with Bluetooth support**, BlueZ cannot
match any profile and returns `br-connection-profile-unavailable`.

**Confirmed fix (tested on OnePlus 6T, Samsung S7 Active, and other dual-mode
targets):** Install `bluez-alsa-utils` on the host:

```bash
sudo apt-get install bluez-alsa-utils
```

This registers the necessary A2DP/audio profile handlers with BlueZ.
Alternatively, ensure PulseAudio or PipeWire is running with its Bluetooth
module loaded.  Run `bleep --check-env` to verify audio tool availability.

> **Note:** This is a **host system dependency**, not a BLEEP bug.  The same
> device may enumerate successfully from a desktop with PulseAudio but fail
> from a headless server without any audio stack.

## Output Format

Both commands produce tree-formatted output via `format_gatt_tree()` (`bleep/ble_ops/conversion.py`).  The tree handles both the lowercase mapping format (standard/non-deep) and the uppercase format (deep).

Example tree structure:

```
── Device: CC:50:E3:B6:BC:A6 (MyDevice)
   ├── Service: 00001800-0000-1000-8000-00805f9b34fb (Generic Access)
   │   ├── Char: 00002a00-0000-1000-8000-00805f9b34fb (Device Name)
   │   │   Handle: 0x0003  Flags: [read]
   │   │   Value: "MyDevice"
   │   │   Raw: [77, 121, 68, 101, 118, 105, 99, 101]
   │   └── Char: 00002a01-0000-1000-8000-00805f9b34fb (Appearance)
   │       Handle: 0x0005  Flags: [read]
   │       Value: null
   │       Raw: [0, 0]
   ...
   ── Landmine Map: { ... }
   ── Permission Map: { ... }
```

When `naggy` change detection flags characteristics, they are annotated with `⚡ CHANGED` indicators in the tree.

## Database Persistence

Both commands persist enumeration results to the BLEEP observation database (`bleep/core/observations.py`):

- **`gatt-enum`**: The CLI handler (`cli.py`) explicitly iterates the mapping, calls `upsert_services()` and `upsert_characteristics()` with FK-defense kwargs (`mac`, `service_uuid`).  Skipped when `--report` is active.
- **`enum-scan`**: Persistence is handled by `_persist_mapping()` in `bleep/ble_ops/scan.py`, which is called after all variants.

UUIDs are normalised to uppercase at the database layer.  See [Observation Database](observation_db.md) for schema details.
